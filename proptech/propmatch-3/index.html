不動産物件データ名寄せ標準化キット「PropMatch」

不動産データ名寄せ実装シナリオガイド

対象読者： 不動産テック担当者・データ管理担当者
目的： 手元から始められる名寄せ実装の全体像を把握する

1. なぜ不動産データは「揺れる」のか

不動産データの揺れは、業界構造に起因する構造的問題である。

原因 具体例
入力者の多様性 仲介・管理・オーナーが各自入力
表記ルール不統一 「丁目」vs「-」、「号室」vs「#」
住所体系の複雑さ 大字・字・通称地名・番地の揺れ
建物名の表記ゆれ 「ハイツ〇〇」「ﾊｲﾂ〇〇」「HEIGHTS〇〇」
新旧データ混在 区画整理・住居表示変更後の旧住所残留
外部データとの突合 SUUMO・HOME'S・自社DB間の形式差異

本質的な問題： 不動産には「唯一の正規ID」が存在しない。登記・住居表示・固定資産税台帳がそれぞれ独立しており、完全統合には行政DBへのアクセスが必要になる。

2. クレンジング優先順位の判断基準

投資対効果マトリクス（物件数 × 揺れ頻度）

揺れ頻度：低（月1件未満） 揺れ頻度：中（月1〜10件） 揺れ頻度：高（月10件超）
物件数：少（〜500件） 優先度4：後回し 優先度3：手動対応 優先度2：テンプレート整備
物件数：中（500〜5,000件） 優先度3：手動対応 優先度2：ツール導入 優先度1：即着手
物件数：多（5,000件超） 優先度2：ツール導入 優先度1：即着手 優先度0：外部基盤検討

判断ルール： - 優先度0〜1：本ガイドの3ステップで対処 - 優先度2〜3：本ガイド＋定期バッチで対処 - 優先度4：対応コスト＞効果のため後回し可

3. 3ステップ実装フロー

ステップ①：テンプレートでルール定義

normalization_rules.csv を作成し、揺れパターンと正規形を定義する。

field,raw_pattern,normalized,priority
building_name,ハイツ|ﾊｲﾂ|HEIGHTS,ハイツ,1
building_name,コーポ|ｺｰﾎﾟ|Corp,コーポ,1
address,番地|番,番地,2
address,丁目|-(?=\d),丁目,2
room_no,号室|#|No\.,号室,3
prefecture,東京都|東京,東京都,1

ルール定義のポイント： - priority が低いほど先に適用（競合回避） - 正規表現可（Pythonの re モジュール想定） - チームで合意したルールをここに集約する

ステップ②：ツールCSVで既存データ判定

check_properties.py（100行以下のPythonスクリプト）で既存データをスキャンする。

入力： properties_raw.csv（既存DB出力）
処理： ルールCSVと照合し、揺れ箇所を検出
出力： check_result.csv

property_id,field,raw_value,matched_rule,suggested_value,confidence
P001,building_name,ﾊｲﾂ山手,building_name_1,ハイツ山手,high
P002,address,渋谷区3-5,address_2,渋谷区3丁目5番地,medium
P003,room_no,101#,room_no_3,101号室,high
P004,building_name,コーポ青山,building_name_1,コーポ青山,skip（既に正規形）
P005,prefecture,東京,prefecture_1,東京都,high

confidence の定義： - high：完全一致 → 自動修正可 - medium：部分一致 → 人間確認推奨 - low：類似のみ → 手動対応

ステップ③：結果CSVで修正指示の自動化

check_result.csv の high 行のみ自動修正スクリプトに渡す。

# apply_fixes.py（概念コード）
for row in result_csv:
    if row['confidence'] == 'high':
        db.update(row['property_id'], row['field'], row['suggested_value'])
        log.write(f"AUTO: {row['property_id']} {row['field']} fixed")
    elif row['confidence'] == 'medium':
        review_queue.append(row)  # 人間レビュー待ち

自動化スコープ： - high：自動適用 → ログ保存 - medium：レビューキューに追加 → 担当者がExcelで確認 - low：週次レポートに集計 → ルール追加を検討

5. 対処範囲の線引き

手元（ローカルツール）で対処できる範囲

  • 全角/半角の統一
  • 建物名の表記ゆれ（正規表現+辞書）
  • 号室フォーマットの統一
  • 都道府県補完（47都道府県辞書）
  • 重複物件の仮マッチング（類似度スコア）

外部API・データ基盤が必要な範囲

課題 必要なもの
正確な住所正規化 国土地理院API、IMI住所変換
登記情報との突合 法務省登記情報サービス（有料）
大規模重複排除（万件超） データ基盤（BigQuery等）＋MLモデル
リアルタイム名寄せ API連携 + ストリーム処理基盤
区画整理・住所変更の追跡 住所変更DBサービス（商用）

6. 用語集

用語 意味
名寄せ 同一物件を指す複数レコードを統合すること
表記ゆれ 同じ内容を異なる形式で記載したもの
正規化 表記ゆれを統一したルールに変換する処理
突合（マッチング） 2つのデータセットで同一物件を対応づけること
文字列距離 2文字列の類似度を測る指標（レーベンシュタイン距離等）
confidence 自動修正の確信度。high/medium/lowで分類
住居表示 市区町村が定める公式の住所体系
